# 치과 웹사이트 로그인 + 예약 관리 시스템 (Firebase 기반 통합 구조)

## 1. 프로젝트 개요

본 프로젝트는 치과 홈페이지의 예약 및 회원 관리를 위한 웹 기반 시스템으로, 일반 사용자, 관리자, 예약 관리 기능을 하나의 서비스 흐름으로 통합한 포트폴리오형 웹 애플리케이션이다.

기존의 정적 치과 홈페이지 구조를 유지하면서, JavaScript와 Firebase를 연동하여 실제 서비스처럼 동작하는 로그인, 회원가입, 예약 신청, 예약 조회, 관리자 관리 기능을 구현한다.

이 문서는 다음 목적을 가진다.

- 사용자 인증 및 권한 분리 구조를 명확히 설명한다.
- Firebase 기반 데이터 저장 흐름을 문서화한다.
- 관리자와 일반 사용자의 역할 차이를 정의한다.
- 실제 구현 파일 구조와 연결되는 설계 문서로 활용할 수 있게 구성한다.

### 2.1 핵심 목표

- 관리자 계정으로 모든 예약/회원 데이터를 관리할 수 있다.

### 2.2 포트폴리오 적합성

- Firebase 인증/데이터베이스 연동 경험

- CRUD 기반 예약 관리 시스템 구현 경험

## 5. 데이터베이스 설계

본 프로젝트는 Firebase Firestore 기반으로 구현되며, 다음 컬렉션을 사용한다.

### 5.1 users 컬렉션

회원 정보를 저장한다.

```text
users/
  {userId}/
    name: "홍길동"
    phone: "010-1234-5678"
    phoneNormalized: "01012345678"
    role: "user"
    createdAt: Timestamp
    updatedAt: Timestamp
```

### 5.2 reservations 컬렉션

예약 정보를 저장한다.

```text
reservations/
  {reservationId}/
    userId: "..."
    name: "홍길동"
    phone: "010-1234-5678"
    phoneNormalized: "01012345678"
    email: "example@domain.com"
    treatmentType: "orthodontics"
    date: "2026-07-01"
    notes: "치아가 아픔"
    status: "pending"
    authCode: "5678"
    createdAt: Timestamp
    updatedAt: Timestamp
```

### 5.3 notices 컬렉션

공지사항 데이터를 저장한다.

```text
notices/
  {noticeId}/
    title: "설 연휴 휴무 안내"
    content: "휴무 안내 내용"
    category: "공지"
    image: ""
    createdAt: Timestamp
```

## 6. 회원가입 시스템 설계

### 6.1 입력 항목

회원가입 시 사용자가 입력하는 정보는 다음과 같다.

- 이름
- 전화번호
- 이메일(선택 또는 확장 가능)

### 6.2 회원가입 처리 흐름

1. 사용자가 회원가입 폼에 입력한다.
2. 입력값 유효성 검사를 수행한다.
3. 전화번호를 정규화한다.
4. Firebase users 컬렉션에 사용자 문서를 생성한다.
5. 저장이 완료되면 로그인 페이지로 이동한다.

### 6.3 회원가입 규칙

- 전화번호는 중복 여부가 중요하다.
- 전화번호는 숫자만 저장되도록 정규화한다.
- 기본 권한은 user로 설정한다.
- 저장 성공 후 로그인 페이지로 이동한다.

### 6.4 보안 고려사항

- 비밀번호는 현재 프로젝트에서 평문 형태로 저장되는 구조이므로, 실제 운영 서비스로 확장 시 Firebase Authentication 연동이 필요하다.
- 포트폴리오 목적상 현재는 간단한 구조로 구현되었다.

## 7. 로그인 시스템 설계

### 7.1 일반 사용자 로그인

일반 사용자는 전화번호 기반으로 로그인한다.

#### 처리 흐름

1. 입력한 전화번호를 Firebase users 컬렉션에서 조회한다.
2. 일치하는 사용자가 있으면 로그인 성공으로 간주한다.
3. 세션 정보를 sessionStorage에 저장한다.
4. 예약 관리 페이지로 이동한다.

### 7.2 관리자 로그인

관리자는 고정 관리자 계정으로 로그인한다.

```text
ID: admin
PASSWORD: 123
```

#### 처리 흐름

1. 입력한 아이디/비밀번호를 하드코딩된 관리자 계정과 비교한다.
2. 일치하면 관리자 세션을 생성한다.
3. 예약 관리 페이지로 이동한다.
4. 관리자 전용 UI가 활성화된다.

### 7.3 세션 저장 방식

로그인 성공 시 sessionStorage에 사용자 정보를 저장한다.

```javascript
sessionStorage.setItem("smileDentalSession", JSON.stringify(session));
```

저장되는 주요 정보는 다음과 같다.

- type: user 또는 admin
- phone
- userId
- name
- loginTime
- permissions

## 8. 예약 관리 시스템 설계

### 8.1 예약 입력 항목

예약 신청 시 사용자가 입력하는 항목은 다음과 같다.

- 이름
- 전화번호
- 이메일
- 진료 종류
- 선호 날짜
- 요청사항

### 8.2 진료 종류

기본적으로 다음 항목을 선택할 수 있다.

- 교정치료
- 심미보철
- 치주질환
- 사랑니

### 8.3 예약 처리 흐름

1. 사용자가 예약 폼을 작성한다.
2. 입력값을 검증한다.
3. Firebase reservations 컬렉션에 저장한다.
4. 저장 성공 시 예약 목록에 반영된다.
5. 사용자 또는 관리자 화면에서 해당 예약을 확인할 수 있다.

### 8.4 예약 상태 관리

예약 데이터는 다음 상태를 가질 수 있다.

- pending: 접수됨
- completed: 완료
- cancelled: 취소

## 9. 사용자 예약 조회 기능

### 9.1 본인 예약 확인

일반 사용자는 본인 예약만 조회할 수 있다.

#### 조회 방식

- 로그인 세션의 userId를 기준으로 reservations를 조회한다.
- userId가 없을 경우 전화번호를 기반으로 보조 조회한다.

### 9.2 본인 인증 방식

현재 구현은 전화번호 기반으로 예약을 조회하는 방식이며, 예약 시 생성되는 인증 코드는 전화번호 뒤 4자리로 처리된다.

```text
인증 번호 = 전화번호 뒷 4자리
```

이 구조는 사용자가 자신이 신청한 예약을 쉽게 확인할 수 있도록 설계되었으며, 향후 실제 서비스로 확장될 경우 Firebase Authentication 또는 별도의 인증 로직으로 보강 가능하다.
